Chapter 1. 컴퓨터 추상화와 기술

1-1. 트랜지스터와 무어의 법칙

  • 트랜지스터 = 스위치. 전류를 흘리거나 끊거나. 단자 3개: 게이트(Gate) / 소스(Source) / 드레인(Drain).
    • 게이트에 전압을 인가 → 소스-드레인 사이 전류량 조절.
    • 전압 세게 주면 채널이 두껍게 생겨 전류 많이 흐름, 작게 주면 적게 흐름 → 스위칭 가능.
  • 트랜지스터 6개를 엮으면 1 bit 저장 가능 (SRAM 셀).
  • 무어의 법칙: 단위 면적당 트랜지스터 수가 약 1.5~2년마다 2배로 증가.
    • 단, 2000년대 이후 사실상 깨졌음. 전력 문제 등 여러 물리적 제약 때문.

1-2. 컴퓨터 응용 분야 (3대 분류) 🔥

구분 특징 비고
PC (개인용 컴퓨터) 범용성(general-purpose) 보유. 제3자 소프트웨어 설치/실행 가능 동영상 플레이어, 브라우저, 편집기 등 자유 설치
서버 (Server) PC보다 성능 ↑. Dependability(신뢰도) 중요. "24/7" 운영 SSH/Telnet 원격 접속. 과거 메인프레임의 현대형. 카카오 서버 마비 사태 등이 dependability 사례
임베디드 (Embedded) 범용성 없음. HW/SW가 강하게 결합. 특정 기능만 수행. 전력·자원 엄격 제한 신호등, IoT 엣지 디바이스 등
  • 슈퍼컴퓨터 = 서버의 일종. 수십~수백 개 노드를 근거리 고속 네트워크(예: InfiniBand) 로 묶음.
    • 2024 기준 1위: El Capitan (코어 약 1,100만 개), 2위 Frontier, 3위 Aurora.
    • 전체 컴퓨터 시장 점유율은 매우 작으나 최근 AI 기반 GPU 슈퍼컴퓨팅팜이 부상.

1-3. SI 단위 vs IEC 이진 단위 🔥

체계 밑 단위 예시
SI (10진) 10 K(10³), M(10⁶), G(10⁹), T(10¹²), P(10¹⁵), E(10¹⁸), Z(10²¹), Y(10²⁴), R(10²⁷), Q(10³⁰)
IEC (2진) 2 Ki(2¹⁰), Mi(2²⁰), Gi(2³⁰), Ti(2⁴⁰), Pi, Ei, Zi, Yi
  • 외워야 하는 최소 범위: Y(요타)까지. Q(쾌카), R까지는 보너스.
  • 왜 중요? SSD 512GB 샀는데 컴퓨터가 더 작게 인식하는 이유 → 판매=SI(10진), 인식=IEC(2진) 이라서.
  • 대문자 B = Byte(바이트, 8bit), 소문자 b = bit. 혼용 금지 (8배 차이).
  • 컴퓨터 내부 연산 단위는 항상 IEC(2진).

1-4. 포스트 PC 시대와 클라우드

  • PMD (Personal Mobile Device): 배터리 동작, 이동 가능, 인터넷 연결 가능. 스마트폰, 태블릿, 스마트 안경 등.
  • 클라우드 컴퓨팅 vs 슈퍼컴퓨터/클러스터 🔥
    • 클라우드: 노드들이 인터넷으로 연결. 물리적으로 원격에 분산(서울-부산-대구 등).
    • 클러스터/슈퍼컴퓨터: 노드들이 근거리 고속 네트워크로 한 공간에 모여 있음.
    • 공통점: 대규모 병렬 처리.
  • 클라우드 서비스 3계층 🔥
    • SaaS (Software as a Service): 소프트웨어 자체 제공. 예) Google Docs, Google Slides.
    • IaaS (Infrastructure as a Service): 가상 컴퓨터 인프라 제공. 예) AWS EC2, Azure.
    • PaaS (Platform as a Service): 플랫폼 제공. 예) 데이터 분석 플랫폼.

1-5. 프로그램 빛의 세계 — SW 계층

┌──────────────────────────────┐
│  Application Software (응용)  │   ← 사용자 프로그램
├──────────────────────────────┤
│  System Software (OS/컴파일러)│   ← 추상화 제공
├──────────────────────────────┤
│  Hardware (CPU, Memory, I/O)  │   ← 실제 기계
└──────────────────────────────┘
  • OS는 두 얼굴: (HW 입장) 시스템 자원 관리자, (User 입장) HW 접근 인터페이스.
  • 컴파일러: 하이레벨 언어(C/Java/Python…) → 어셈블리 언어 → 기계어.
  • 기계어 = 0과 1로 표현되는 실제 명령 비트열이다.
  • 어셈블리 ↔ 기계어 = 1:1 매핑. 어셈블리는 사람이 읽기 쉬운 형태(mnemonic).

1-6. 컴퓨터의 5대 구성 요소

  1. 입력 (Input)
  2. 출력 (Output)
  3. 메모리 (Memory) — 명령어와 데이터 저장
  4. 데이터패스 (Datapath) — 데이터가 이동하는 경로 (ALU, 레지스터 연결 등)
  5. 제어 (Control Unit) — 명령어 해석 및 제어 신호 생성
  • ALU (Arithmetic Logic Unit): 산술 연산(+−×÷) + 논리 연산(크기 비교 등) 담당.
  • Datapath: 한 번에 1 word 폭으로 병렬 전송한다. 패러럴은 근거리용, 시리얼은 원거리용(스큐/드리프트 때문)이라는 감각을 잡아두면 된다.

1-7. iPhone A12 SoC 해부

  • SoC (System on Chip): 칩 하나 안에 시스템 전체가 내장된 집적회로(Integrated Circuit).
  • A12 내부: CPU 6개 코어(빅 2 + 리틀 4) + GPU + NPU(Neural Processing Unit) + 메모리 컨트롤러.
  • big.LITTLE 아키텍처 🔥
    • Big Core: 고성능. 무거운 작업 담당.
    • Little Core: 저성능·저전력. 가벼운 백그라운드 작업.
    • 목적 = 전력 소모 최소화 (필요할 때만 Big Core 가동).
  • PMIC (Power Management IC): 전원 관리 칩. 가전제품 전원 고장 시 십중팔구 PMIC 문제.
  • Fabless: 생산 공장 없이 설계만 파는 회사 (예: ARM). 삼성 Exynos도 내부는 ARM 기반.

1-8. 메모리 종류: DRAM vs SRAM 🔥

항목 SRAM DRAM
구성 트랜지스터 6개 / bit 트랜지스터 1개 + 커패시터 1개 / bit
속도 빠름 느림
전력 상시 공급 필요 (많음) 주기적 Refresh만 필요 (적음)
집적도 낮음 높음
가격 비쌈 쌈
용도 캐시(L1/L2/L3) 메인 메모리
  • Static/Dynamic은 속도가 아니라 "저장 메커니즘"을 뜻함. (Static=전기 계속 공급, Dynamic=커패시터+주기적 리프레시)
  • DRAM 커패시터는 절연체로 감싸져 있지만 전하가 조금씩 누출 → Refresh 필요.

1-9. 메모리 계층 구조 🔥

[빠름/작음/비쌈]
    Register     수 KB        ← CPU 클럭 1사이클에 접근
    L1 Cache     수십 KB
    L2 Cache     수백 KB ~ MB
    L3 Cache     수 MB
    Main Memory (DRAM)  GB    ← ns 단위
    SSD           수백 GB ~ TB ← μs 단위
    HDD           TB           ← ms 단위 (SSD보다 약 1000배 느림)
    Cloud/Network Storage     ← 이론상 무한
[느림/큼/쌈]
  • 각 계층은 약 1000배씩 속도/크기 차이.
  • 잘 동작하는 이유 = Locality(지역성) 🔥
    • Temporal Locality (시간 지역성): for문처럼 같은 코드 반복 실행.
    • Spatial Locality (공간 지역성): 배열 순차 접근처럼 인접 메모리 참조.
  • 이상적 메모리(빠름+큼+쌈+저전력+비휘발성)는 존재할 수 없음 → 절충안이 계층 구조.

1-10. 휘발성 vs 비휘발성 메모리

  • Volatile (휘발성): 전원 끄면 데이터 사라짐. Register, SRAM, DRAM.
  • Non-volatile (비휘발성) / Secondary Memory: 전원 꺼져도 유지. HDD, SSD(플래시), 광학 디스크.
  • ※ 비휘발성이라고 100% 안전한 건 아님 (SSD 컨트롤러 고장 등).

1-11. 반도체 칩 생산 공정

실리콘 잉곳 → 슬라이서 → 웨이퍼 → 20~40단계 공정(도핑/에칭/메탈 등)
    → 웨이퍼 테스트 → 다이싱(자르기) → 패키징 → 최종 테스트 → 출하
  • 웨이퍼: 보통 직경 300mm(12인치). 동그란 원판.
  • 다이(Die): 웨이퍼를 잘라낸 개별 칩 단위. 네모남.
  • Yield(수율) = 정상 다이 비율. 수율이 수익성 좌우.
  • 공정(예: 10nm): 소자를 제어할 수 있는 최소 단위 폭. 작을수록 집적도 ↑.
  • 집적회로 발전: IC → LSI → VLSI → ULSI.

1-12. 반도체 기본 원리 (개념 수준만)

  • 주성분은 규소(Si). 외곽 전자 4개로 안정적 (기본적으로 절연).
  • 도핑 (Doping): 불순물을 섞어 전기 특성 부여.
    • N형: 5가 원소(예: 인) 주입 → 자유 전자 남음 → (−) 성질.
    • P형: 3가 원소(예: 붕소) 주입 → 정공(hole) 생김 → (+) 성질.
  • P-N 접합 = 다이오드 (전류 한 방향).
  • NPN / PNP 접합 = 트랜지스터. 게이트 전압으로 채널 두께(전류량) 조절.
  • CMOS: Complementary Metal-Oxide-Semiconductor. 현재 반도체 제조 기술의 표준.

1-13. 성능(Performance) — 기초

  • "성능이 좋다"는 표현 금지. 어떤 지표(metric)냐가 중요.
    • 비행기 비교 예시: 승객 수(A380 최고), 순항거리(777 최고), 속도(콩코드 최고), 승객 처리량(A380 최고) — 지표마다 정답 다름.
  • 응답 시간 (Response Time / Execution Time): 작업 개시~종료. 작을수록 좋음.
  • 처리량 (Throughput / Bandwidth): 단위시간당 처리하는 작업량. 클수록 좋음.

1-14. Wall Clock Time vs CPU Time 🔥

  • Wall Clock Time (경과 시간): 실제 벽시계 시간. I/O, OS 오버헤드, 다른 프로세스 간섭 모두 포함.
  • CPU Time: 순수하게 이 프로그램이 CPU를 사용한 시간.
    • = User CPU Time + System CPU Time
    • User Time: 응용 코드(예: for문) 실행 시간.
    • System Time: 시스템 콜(예: fopen→create) 때문에 OS 커널이 대신 수행한 시간.
  • 컴퓨터 구조에서는 CPU Time만 관심 대상.

1-15. 클럭 (Clock) 🔥

  • Clock Period (주기): 한 클럭이 걸리는 시간. 예) 250 ps = 0.25 ns.
  • Clock Rate (Frequency): 1초에 클럭 몇 번? 주기의 역수. 예) 0.25 ns의 역수 = 4 GHz.
  • 외워둘 것: 250 ps ↔ 4 GHz, 500 ps ↔ 2 GHz.
  • 클럭 신호는 방형파(구형파) 사용 (5V↔0V 급격 전환).

1-16. CPU 실행 시간 공식 🔥🔥

CPU Time = (Instruction Count) × (CPI) × (Clock Cycle Time)
         = (Instruction Count) × (CPI) / (Clock Rate)
  • CPI (Cycles Per Instruction): 명령어 하나 처리에 필요한 평균 클럭 사이클 수.
  • 성능 향상 방법 🔥
    1. 명령어 수(IC) 줄이기 — 컴파일러/알고리즘 최적화.
    2. CPI 줄이기 — 아키텍처 개선.
    3. Clock Rate 올리기 — 단, 전력·발열 한계 있음.

예제 1. 같은 프로그램, 다른 컴퓨터

  • 컴퓨터 A: 2 GHz, 10초 걸림 → 총 사이클 = 2G × 10 = 20G 사이클
  • 컴퓨터 B: 6초 안에 끝내야 함. A보다 사이클 1.2배 소비 → 24G 사이클
  • → Clock Rate = 24G / 6 = 4 GHz 필요
  • 교훈: 성능 1.67배 개선에 클럭은 2배 올려야 함 (CPI 증가 때문).

예제 2. 평균 CPI 계산

명령어 타입 CPI Code1 횟수 Code2 횟수
A 1 2 4
B 2 1 1
C 3 2 1
  • Code1: 총 사이클 = 2×1 + 1×2 + 2×3 = 10, 명령어 5개 → CPI = 2.0
  • Code2: 총 사이클 = 4×1 + 1×2 + 1×3 = 9, 명령어 6개 → CPI = 1.5
  • 명령어 개수가 많아도 총 사이클은 더 적을 수 있음 → 단순히 명령어 수만 보면 안 됨.

1-17. 전력 장벽 (Power Wall) 🔥

동적 전력 공식 🔥🔥

Dynamic Power ∝ (1/2) × C × V² × f

  C = Capacitive Load (용량성 부하, 공정 미세화로 감소)
  V = Voltage (전압)
  f = Switching Frequency (스위칭 빈도 = 클럭)

계산 예제

  • 새 CPU: C가 구형의 85%, V도 15% 감소, f도 15% 감소
  • 새 동전력 / 구 동전력 = 0.85 × 0.85² × 0.85 = 0.85⁴ ≈ 0.52
  • 즉 새 CPU는 절반 수준의 전력.

Power Wall

  • 클럭을 계속 높이면 발열이 감당 불가 → 주파수 정체 (최근 CPU ~ 5 GHz 근처에서 멈춤).
  • 역사: 82년 3.3W → 2018년 커피레이크 95W, 클럭 12.5 MHz → ~4 GHz (약 1000배).

해결책 = 멀티코어 + DVFS

  • 싱글 스레드 성능 증가 정체 → 코어 수 증가 방향으로 전환.
  • ILP → TLP 전환:
    • ILP (Instruction-Level Parallelism): 한 스레드 내 명령어 병렬. (Pipeline, Superscalar, Dynamic Scheduling)
    • TLP (Thread-Level Parallelism): 여러 스레드 동시 실행(멀티코어).
  • DVFS (Dynamic Voltage & Frequency Scaling): 부하 따라 전압·주파수 동적 조정.
  • 정전력(static power)은 누설 전류가 일정하다고 보면 P_static = V × I_leak 꼴이므로 전압에 비례한다고 둘 수 있다. 이 가정은 전력 계산 문제에서 자주 사용된다.

1-18. ILP 기법 3가지 🔥

  1. Pipelining: 명령어 실행을 단계별로 겹쳐 처리하는 방식이다. 전형적인 5단계 예시로 IF → ID → EX → MEM → WB 같은 형태를 떠올리면 된다.
  2. Superscalar: 한 사이클에 여러 명령어를 다중 파이프라인으로 동시 실행.
  3. Dynamic Instruction Scheduling: 하드웨어가 런타임에 명령어 종속성 검사해 순서 재배치.

1-19. 7가지 위대한 아이디어 🔥

  1. Abstraction (추상화) — 설계 단순화
  2. Common Case Fast — 자주 생기는 일을 빠르게
  3. Parallelism (병렬성) — 성능 향상
  4. Pipelining — 병렬성 특수 형태
  5. Prediction (예측) — 분기 예측 등
  6. Memory Hierarchy (메모리 계층)
  7. Redundancy via Dependability (여유분을 이용한 신용도 개선) — 백업, 중복 자원

1-20. 암달의 법칙 (Amdahl's Law) 🔥🔥 시험 필출

T_improved = T_unaffected + (T_affected / Improvement)
  • 특정 파트만 N배 개선해도 전체 성능은 N배 개선되지 않음 (병렬 불가 부분 때문).

예제

  • 전체 실행 시간 100초, 곱셈에 80초 소요. 전체를 4배 빠르게 (100→25초) 하려면?
  • 25 = (100 - 80) + 80 / x → 5 = 80/x → x = 16배. 곱셈을 16배 빠르게 해야 함.
  • 즉 곱셈을 80초 → 5초로 줄여야 전체가 4배 개선됨.

1-21. 흔한 오해들

  • Low Utilization ≠ Low Power
    • i7 CPU: 100% 부하 258W, 10% 부하에서도 121W (47% 소비).
    • 정적 전력(static power) 때문. 안 쓰면 끄는 게 최선.
  • MIPS (Millions of Instructions per Second) 지표 함정 🔥
    MIPS = Clock Rate / (CPI × 10⁶)
    • 동일 CPU라도 컴파일러에 따라 CPI 달라져 MIPS 달라짐 → 절대적 성능 지표로 부적합.
