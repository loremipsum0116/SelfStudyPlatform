2-4. MIPS 명령어 형식 🔥🔥

MIPS는 32비트 고정 길이. 3가지 포맷.

(1) R-Type (Register) — 산술/논리 연산

| op (6) | rs (5) | rt (5) | rd (5) | shamt (5) | funct (6) |
  • op: Operation code.
  • rs: 첫 번째 소스 레지스터.
  • rt: 두 번째 소스 레지스터.
  • rd: 목적지 레지스터 (결과 저장).
  • shamt: Shift Amount (시프트 연산용).
  • funct: op가 0일 때 실제 연산 종류 결정.

예) add $t0, $s1, $s2 → $s1 + $s2 → $t0

  • op=0, funct=32(add), funct=34(sub)
  • shamt=0 (시프트 아님)

레지스터가 5비트인 이유: 2⁵ = 32개 레지스터를 표현하기 위함.

(2) I-Type (Immediate) — 즉시값 사용

| op (6) | rs (5) | rt (5) | constant or address (16) |
  • constant: addi의 경우 상수값으로 해석.
  • address: lw, sw의 경우 오프셋으로 해석.
  • beq, bne의 경우 분기 오프셋.

중요: 16비트뿐이므로 ±2¹⁵ 범위의 값만 표현 가능. 큰 상수는 lui + ori 조합.

예) addi $t0, $t1, 4 → op=8, rs=$t1(9), rt=$t0(8), imm=4

(3) J-Type (Jump) — 무조건 분기

| op (6) | address (26) |
  • j 명령어. op=2.
  • 26비트는 Word 단위 주소 → 실제 바이트 주소 = 26비트 << 2 (왼쪽 시프트 2 = ×4).
  • 총 28비트 + PC의 상위 4비트 그대로 → 32비트 완성.

레퍼런스 카드와 바로 연결해서 보기

형식 대표 명령어 결과/목적지 시험 포인트
R-Type add, sub, and, or, nor, slt, sll, srl, jr 보통 rd op=0이고 실제 연산은 funct가 결정. 시프트는 shamt를 꼭 같이 본다.
I-Type addi, andi, ori, slti, lw, sw, lb, lbu, lh, lhu, beq, bne, lui 보통 rt 또는 분기 오프셋 같은 16비트 immediate라도 sign extension/zero extension이 갈린다.
J-Type j, jal PC 갱신, jal은 $ra도 갱신 26비트 address에 하위 00이 붙고, 상위 4비트는 현재 PC에서 가져온다.

2-5. MIPS 레지스터 32개 🔥🔥🔥 반드시 암기

번호 이름 용도 규칙
$0 $zero 항상 0 (read-only) 상수 0
$1 $at 어셈블러 예약 사용 금지
$2~$3 $v0, $v1 함수 return 값
$4~$7 $a0~$a3 함수 인자 (최대 4개)
$8~$15 $t0~$t7 Temporary (임시) Caller-saved
$16~$23 $s0~$s7 Saved (변수) Callee-saved
$24~$25 $t8, $t9 추가 Temporary Caller-saved
$26~$27 $k0, $k1 OS/커널 예약
$28 $gp Global Pointer (전역변수 시작)
$29 $sp Stack Pointer
$30 $fp Frame Pointer
$31 $ra Return Address jal이 여기에 PC+4 저장

숫자 ↔ 이름 빠른 외우기

  • $t0 = 8, $s0 = 16, $t8 = 24 — 8, 16, 24 기준으로 +순번.
  • $a0 = 4, $v0 = 2.

Caller-saved vs Callee-saved 🔥

  • $t 레지스터 (Caller-saved, "호출하는 애가 저장")
    • 누구나 자유롭게 덮어씀. 함수 호출 전에 호출자가 직접 백업해야 함.
    • 호출된 함수는 $t를 맘대로 써도 됨.
  • $s 레지스터 (Callee-saved, "호출된 애가 저장")
    • 다른 애의 중요한 값이 들어있을 수 있으므로 호출된 함수가 사용 전 백업, 리턴 전 복원.

→ 이 규칙이 없으면 양쪽 다 매번 백업해서 메모리 접근 횟수 폭증(≈1000배 느림).

2-6. 메모리 피연산자 (Load/Store)

Load/Store 명령어

  • lw $t0, offset($s1) — 메모리[$s1 + offset]의 Word(4B) → $t0
  • sw $t0, offset($s1) — $t0 → 메모리[$s1 + offset]
  • lb / lbu — 1바이트 로드. lb는 sign extension, lbu는 zero extension.
  • lh / lhu — 2바이트(Halfword) 로드. lh는 sign extension, lhu는 zero extension.
  • sb / sh — 저장 시에는 각각 하위 8비트, 하위 16비트만 메모리에 기록.
  • ll / sc — 레퍼런스 카드에 같이 실린 원자적(atomic) test-and-set pair. 병렬/동기화 개념을 배울 때 다시 등장한다.

크기별로 한 번에 정리

크기 Load Store 확장 규칙
Byte (1B) lb, lbu sb lb=sign extension, lbu=zero extension
Halfword (2B) lh, lhu sh lh=sign extension, lhu=zero extension
Word (4B) lw sw 32비트 그대로

배열 접근 🔥

  • C: int A[8] → A[8] 접근 시
  • MIPS: 주소 계산 시 인덱스 × 4 필요 (int = 4B이므로).
# g = h + A[8];  에서 A는 $s3(베이스), h는 $s2
lw   $t0, 32($s3)     # 32 = 8 * 4
add  $s1, $s2, $t0    # $s1 = g

Memory Alignment 🔥

  • MIPS는 Word Alignment 요구.
  • Word 시작 주소는 항상 4의 배수.
  • 주소 하나당 1 Byte 저장. lw가 가리키는 주소는 반드시 0x...00, 0x...04, 0x...08, ... 식.

레퍼런스 카드식 정렬 기준

데이터 크기 정렬 조건 주소 하위 비트 형태
Byte 제한 없음 xxx
Halfword 2의 배수 주소 xx0
Word 4의 배수 주소 x00
Doubleword 8의 배수 주소 000

즉 바이트 주소의 맨 아래 몇 비트가 0이어야 하느냐만 봐도 정렬 여부를 빠르게 판정할 수 있습니다.

Endian 🔥🔥

  • MSB를 어디에 저장하느냐가 기준.
방식 MSB 위치 예시 CPU
Big Endian 낮은 주소에 MSB MIPS (대표), 네트워크 프로토콜
Little Endian 낮은 주소에 LSB x86(Intel)
Bi-Endian 설정 가능 ARM

예) 레지스터 값 0x12345678을 주소 100에 저장:

  • Big Endian: 100:12, 101:34, 102:56, 103:78
  • Little Endian: 100:78, 101:56, 102:34, 103:12

→ 좋고 나쁨의 문제 아님. 통일만 되면 됨.

2-7. 상수/즉시값 피연산자

addi — Immediate 값을 명령어에 박음 🔥

addi  $s3, $s3, 4     # $s3 = $s3 + 4
  • 상수를 매번 메모리에서 읽어오는 방식보다 ≈1000배 빠름.
  • 벤치마크상 산술 연산의 50% 이상이 immediate 사용.

$zero 활용 트릭

add  $t2, $s1, $zero    # == move $t2, $s1  (값 복사)

32비트 상수 만들기 — lui + ori 🔥🔥

Immediate 필드는 16비트뿐이라 32비트 상수를 직접 못 담음.

lui  $t0, 0x1234      # 상위 16비트에 0x1234 로드. 하위는 0
ori  $t0, $t0, 0x5678 # 하위 16비트에 0x5678 OR
# 결과: $t0 = 0x12345678
  • lui (Load Upper Immediate): immediate를 레지스터의 상위 16비트에 로드, 하위는 0.

2-8. 논리 연산

Shift 연산 🔥🔥

명령어 의미 빈 자리 채우기
sll Shift Left Logical 0으로
srl Shift Right Logical 0으로 (부호 무시)
sra Shift Right Arithmetic 기존 MSB로 (부호 유지)
sllv/srlv/srav Variable 버전 시프트 양이 레지스터에
  • Arithmetic Shift Left는 없음 (sll로 충분. 왼쪽 시프트는 MSB 자리가 어차피 날아감).
  • 왼쪽 i비트 시프트 = × 2ⁱ
  • 오른쪽 i비트 시프트 = ÷ 2ⁱ (단, 음수는 반드시 sra 써야 함)

음수의 올바른 나눗셈 예시

  • 1110 (= −2) 를 srl로 오른쪽 시프트 → 0111 (= +7) ❌ 엉뚱한 양수.
  • sra로 시프트 → 1111 (= −1) ✓ 정상.

Circular Shift (Rotate)

  • rol/ror: 빠져나간 비트를 반대쪽에 채움.

Bitwise 논리 연산

  • and, or, xor, nor (R-type)
  • andi, ori, xori (I-type)

연산별 용도 🔥

연산 용도
AND 특정 비트 유지/초기화 (마스크). 0과 AND → 0 (초기화), 1과 AND → 원래 값 유지
OR 특정 비트 세팅. 1과 OR → 1로 세팅
NOT 비트 반전. MIPS엔 NOT 명령어 없음 → nor로 구현

NOT을 NOR로 구현 🔥

nor  $t0, $t1, $zero   # $t0 = ~$t1
# 이유: NOR(x, 0) = NOT(x OR 0) = NOT(x)

2-9. 분기(Branch) 연산

기본 분기 명령어 🔥

명령어 의미
beq rs, rt, L $rs == $rt 이면 L로 분기
bne rs, rt, L $rs != $rt 이면 L로 분기
j L 무조건 L로 점프
jr $ra $ra의 값으로 PC 세팅 (return 등)
jal L L로 점프하면서 PC+4 → $ra 저장 (함수 호출)

if-else 컴파일 예시

if (i == j) f = g + h;
else f = g - h;

bne  $s3, $s4, Else   # i != j 면 Else로
add  $s0, $s1, $s2    # f = g + h
j    Exit
Else: sub $s0, $s1, $s2
Exit: ...

Set-Less-Than 계열

  • slt rd, rs, rt: $rs < $rt 면 $rd=1, 아니면 0 (signed)
  • sltu: unsigned 비교
  • slti/sltiu: immediate 버전

Signed vs Unsigned 비교 🔥

  • slt (signed): 산술 연산용. 음수 있는 값.
  • sltu (unsigned): 주소 계산, 논리적 크기 비교.

예) $s0 = 0xFFFFFFFF, $s1 = 0x00000001

  • slt $t0, $s0, $s1 → $s0=−1 < $s1=1 → $t0 = 1
  • sltu $t0, $s0, $s1 → $s0=거대수 > $s1=1 → $t0 = 0

빠른 경계 검사 (Fast Bound Check) 🔥

if (0 <= i < n) { ... }

를 한 번의 sltu로 해결:

sltu $t0, $s0, $t1   # $s0=i, $t1=n
beq  $t0, $zero, Error
  • 원리: i가 음수면 unsigned 해석 시 거대한 값이 되어 n보다 크게 됨 → 한 번에 검증 가능.

2-10. 의사 명령어 (Pseudo Instructions) 🔥

  • 실제 하드웨어에 존재하지 않지만 어셈블러가 여러 실제 명령어로 풀어주는 가짜 명령어.
  • 프로그래밍 편의상 제공.
의사 명령어 실제 동작
li $t0, 5 addi $t0, $zero, 5
move $t0, $t1 add $t0, $t1, $zero
blt $t0, $t1, L slt $at, $t0, $t1 + bne $at, $zero, L
la $t0, label lui + ori 조합

레퍼런스 카드에는 특히 blt, bgt, ble, bge, li, move가 따로 정리되어 있습니다. 시험에서도 이 여섯 개는 "실제 명령어가 아니라 어셈블러가 풀어쓴다"는 점을 자주 확인합니다.

왜 실제 명령어로 안 만들었나? 🔥

  • blt를 실제 명령어로 만들면 비교 + 분기가 한 명령어에 담겨 사이클이 길어짐.
  • CPU는 최악의 명령어 기준으로 클럭 사이클이 정해지므로, 다른 명령어까지 느려짐.
  • → 짧은 명령어 2개가 전체 성능에 더 유리.
  • ※ 일부 어셈블러는 의사 명령어를 지원 안 할 수도 있으므로 실제 명령어로 쓰는 것도 필요.

2-11. 기본 블록 (Basic Block)

컴파일러 최적화의 기본 단위.

기본 블록의 정의:

  1. 중간에 분기(branch/jump) 명령어가 없음 (단, 맨 끝은 예외).
  2. 중간에 분기 타겟(label) 이 없음 (단, 맨 처음은 예외).

→ 입구 1개, 출구 1개, 중간은 무조건 순차 실행. "신호등 없는 직선 도로".

2-12. 프로시저 호출 (Procedure Call) 🔥🔥

프로시저 호출 6단계

  1. 매개변수를 레지스터($a0~$a3)에 저장 (4개 초과 시 스택 이용)
  2. 제어를 프로시저로 이동 (PC ← procedure 주소)
  3. 로컬 변수용 저장 공간 확보 (스택에)
  4. 프로시저 내부 작업 수행
  5. 반환값을 $v0, $v1에 저장
  6. 호출한 위치로 복귀 (PC ← $ra)

관련 명령어

  • jal ProcName: Jump and Link
    • PC + 4 → $ra (복귀 주소 저장)
    • PC ← ProcName
  • jr $ra: Jump Register. $ra → PC (복귀)

스택 메모리 구조 🔥

High Address (0x7FFFFFFC)
┌────────────────┐
│  Stack         │  ← $sp (아래로 자람)
│     ↓          │
│                │
│     ↑          │
│  Heap          │  ← malloc 영역
├────────────────┤
│  Static Data   │  ← 전역 변수 ($gp가 가리킴)
├────────────────┤
│  Text (code)   │  ← 명령어. PC가 이 영역을 돌아다님
├────────────────┤
│  Reserved      │
└────────────────┘
Low Address (0x00000000)
  • MIPS 32비트: 스택은 0x7FFFFFFC에서 아래로 자람.
  • Stack Frame (Activation Record): 함수 호출마다 쌓이는 블록.
    • 포함 내용: Saved $ra, Saved Arguments, Saved $s Registers, 지역변수
  • $fp: 현재 프레임의 베이스 (고정), $sp: 현재 프레임 최상단 (변동).

레퍼런스 카드 그림 기준으로 본 전형적인 주소 배치

  • Text 영역은 보통 0x00400000 근처에서 시작.
  • Static Data는 0x10000000 근처.
  • Heap(동적 데이터)는 그 위쪽, 예시 카드에선 0x10008000 근처부터 자라기 시작.
  • Stack은 위쪽 높은 주소에서 아래쪽으로 성장.
Higher addresses
┌──────────────────────┐
│ Argument 6, 5, ...   │
├──────────────────────┤  ← $fp 근처
│ Saved Registers      │
├──────────────────────┤
│ Local Variables      │
├──────────────────────┤  ← $sp
│   더 아래 주소 ...    │
└──────────────────────┘
Lower addresses

핵심은 $fp는 프레임 안의 기준점, $sp는 현재 쌓인 스택의 끝이라는 점입니다. 그래서 지역변수/저장 레지스터 오프셋 계산은 보통 $fp 기준으로 설명하면 더 안정적으로 이해됩니다.

Leaf Procedure 예제 (함수가 함수를 호출 안 함)

int leaf(int g, int h, int i, int j) {
    int f = (g + h) - (i + j);
    return f;
}

leaf:
    addi $sp, $sp, -4         # 스택 4바이트 확보 (f용)
    sw   $s0, 0($sp)          # 호출자의 $s0 백업
    add  $t0, $a0, $a1        # g + h
    add  $t1, $a2, $a3        # i + j
    sub  $s0, $t0, $t1        # f
    add  $v0, $s0, $zero      # return value → $v0
    lw   $s0, 0($sp)          # $s0 복원
    addi $sp, $sp, 4          # 스택 정리
    jr   $ra                  # 복귀

Non-Leaf (중첩) 프로시저 🔥

  • 자신이 또 다른 프로시저를 호출 (재귀 포함).
  • 호출 전: $ra, 사용 중인 $a, $s, 지역변수 모두 스택에 저장 필요.
  • 호출 후: 스택에서 복원.

Factorial 예제 (재귀)

int fact(int n) {
    if (n < 1) return 1;
    else return n * fact(n-1);
}

핵심 흐름:

  1. $sp를 8 감소 (Return Address + Argument 저장 공간).
  2. $ra와 $a0(n)를 스택에 저장.
  3. n < 1 체크.
  4. $a0를 1 감소시키고 jal fact → 재귀.
  5. 복귀 후, 저장했던 $ra와 n 복원.
  6. n * fact(n-1) 계산 → $v0.
  7. $sp 복원, jr $ra.

→ 스택에 프레임이 계속 쌓이다가 return 시 하나씩 정리.

2-13. 문자와 문자열

문자 인코딩

  • ASCII: 7비트. 128개 문자. 'A'=65, 'a'=97, '0'=48 정도는 암기.
  • Latin-1: 8비트, 서유럽권.
  • Unicode / UTF-8: 전 세계 문자 (한글 깨짐은 여기서 발생).

레퍼런스 카드 기준 ASCII 최소 암기 세트

문자 10진 16진 왜 자주 보나
NUL 0 0x00 C 문자열 종료 문자 '\0'
LF 10 0x0A 줄바꿈
CR 13 0x0D 캐리지 리턴
Space 32 0x20 공백 문자
'0' ~ '9' 48 ~ 57 0x30 ~ 0x39 숫자 문자는 연속 배치
'A' ~ 'Z' 65 ~ 90 0x41 ~ 0x5A 대문자 범위
'a' ~ 'z' 97 ~ 122 0x61 ~ 0x7A 소문자 범위

즉 문자와 숫자는 결국 비트 패턴이고, 바이트 로드/스토어 명령어가 필요한 이유도 바로 여기에 있습니다.

바이트 단위 처리 명령어 🔥

  • ASCII는 7비트라 워드로 다루면 낭비 → 바이트 단위 접근 필요.
  • lb (Load Byte, signed ext), lbu (Load Byte Unsigned, zero ext)
  • sb (Store Byte)
  • lh/lhu/sh: Half-word 버전.

strcpy 예제

void strcpy(char x[], char y[]) {
    int i = 0;
    while ((x[i] = y[i]) != '\0') i++;
}

→ lbu로 y[i] 읽고 → sb로 x[i]에 저장 → null이면 종료.

strcpy:
    addi $sp, $sp, -4
    sw   $s0, 0($sp)
    add  $s0, $zero, $zero   # i = 0
L1: add  $t1, $s0, $a1       # y + i
    lbu  $t2, 0($t1)         # y[i]
    add  $t3, $s0, $a0       # x + i
    sb   $t2, 0($t3)         # x[i] = y[i]
    beq  $t2, $zero, L2      # null이면 종료
    addi $s0, $s0, 1
    j    L1
L2: lw   $s0, 0($sp)
    addi $sp, $sp, 4
    jr   $ra

2-14. MIPS 주소 지정 방식 (Addressing Modes) 🔥🔥

  1. Immediate Addressing — 상수 값 자체가 명령어에 포함 (addi).
  2. Register Addressing — 피연산자가 레지스터 (add, jr).
  3. Base/Displacement Addressing — 베이스 레지스터 + offset (lw, sw).
  4. PC-Relative Addressing — 조건부 분기 (beq, bne).
  5. Pseudo-Direct Addressing — 점프 명령어 (j).

2-15. 분기 주소 계산 🔥🔥🔥 시험 필출

PC-Relative (beq, bne)

  • 명령어 내 16비트 offset은 Word 단위.
  • 실제 분기 주소:
    Target = (PC + 4) + (offset × 4)
  • 왜 PC + 4인가? CPU는 명령어 Fetch 단계에서 이미 PC를 +4 증가시켜 놓음.
  • 표현 범위: PC 기준 약 ±2¹⁷ 바이트.

Pseudo-Direct (j)

  • 명령어 내 26비트 address도 Word 단위.
  • 실제 주소:
    Target = { PC[31:28] , address , 00 }
    • 상위 4비트: 현재 PC에서 그대로 복사.
    • 중간 26비트: 명령어의 address 필드.
    • 하위 2비트: 00 (Word 정렬이므로).

예제 1. beq 분기 거리 계산

80000: sll  ...
80004: add  ...
80008: lw   ...
80012: bne  $t0, $s5, Exit   ← 여기서 bne
80016: ...
80020: ...
80024: Exit: ...
  • Exit은 현재 명령어 기준 12바이트(3워드) 아래.
  • 실제 offset 값(명령어에 들어갈 값) = 2 (bne 자신 다음부터 셈).
  • 이유: bne 수행 시 PC는 이미 80016(=PC+4). Exit(80024)과의 차이 = 8 = 2 word.

예제 2. j 명령어 주소 필드

  • 레이블 Loop가 메모리 주소 80000에 있으면
  • j 명령어에 들어갈 값 = 80000 / 4 = 20000 (Word 주소).

먼 곳으로 분기 트릭 🔥

beq는 ±2¹⁷ 바이트 범위만 가능. 더 멀리 가고 싶다면:

# beq $s0, $s1, FAR_LABEL  (불가)

bne  $s0, $s1, L2   # 조건 뒤집기
j    FAR_LABEL       # j는 32비트 근처까지 가능
L2: ...

2-16. 명령어 인코딩/디코딩 변환 🔥🔥 시험 필출

변환 절차 (16진수 → assembly)

  1. 16진수 → 2진수 (4비트씩).
  2. 앞 6비트 (op) 로 타입 식별.
  3. 타입별로 필드 분해.
  4. 레지스터 번호를 이름으로 변환.

예제: 0x00A83020

0x00A83020 = 0000 00|00 101|0 1000|0 0110|0 000|00 100000

op = 000000 (=0) → R-Type
rs = 00101  (=5)  → $a1
rt = 01000  (=8)  → $t0
rd = 00110  (=6)  → $a2
shamt = 00000 (=0)
funct = 100000 (=32) → add

→ add $a2, $a1, $t0 — ※ 교재 예시에 따라 순서 혼동 주의.

주요 op/funct 값 암기 포인트 🔥

명령어 타입 op funct
add, sub, and, or, sll, srl, sra R 0 add=32, sub=34, sll=0, srl=2, sra=3
addi I 8 —
lw I 35 —

2-17. 마지막 체크 — 오류 & 함정

  1. "강력한 명령어가 무조건 빠르다?" ❌
    • 복잡한 명령어는 사이클이 길어져 전체 클럭을 끌어내림.
  2. "어셈블리로 직접 짜면 최고 성능?" ❌
    • 요즘 컴파일러는 매우 정교. 꼭 그렇지 않음.
  3. "상업용 소프트웨어의 호환성 = 명령어 집합이 불변" ❌
    • 아키텍처는 계속 진화. 단, 하위 호환성(backward compatibility)은 대부분 유지됨.
  4. 분기 오프셋 단위 혼동 🔥
    • J 명령어의 26비트, 분기 명령어의 16비트 모두 Word 단위. 실제 바이트로는 ×4.
  5. PC-Relative 계산 시 PC+4 빼먹기 금지 🔥
    • 이미 PC는 +4 되어 있다는 사실이 핵심.
