전제 사실 🔥

  • 곱셈 > 나눗셈 (곱셈이 훨씬 빠름). 이유: 곱셈은 병렬화·파이프라이닝 가능, 나눗셈은 순차적.
  • 2의 지수승 곱/나누기는 Shift로 처리 → ALU 연산 불필요.
  • 해시 펑션 등 설계 시 곱셈 기반 > 나눗셈 기반이 성능상 유리.

3-1. 덧셈과 뺄셈 🔥

기본 원리

  • 2진수 덧셈은 10진수 덧셈과 동일. 단, 자리마다 합이 2가 되면 Carry(올림수) 발생.
  • 뺄셈은 직접 회로로 만들지 않음 → 2의 보수 취해서 덧셈으로 처리.
    • 이유: 2의 보수는 NOT 게이트 + 1만으로 HW 구현 가능 → 매우 저렴.

덧셈에서의 Overflow 🔥🔥

  • 부호가 다른 두 수를 더할 때는 Overflow 불가능. (양수+음수 → 범위 안)
  • Overflow 발생 케이스 2가지:
케이스 결과 판정
양수 + 양수 = 음수 MSB가 0+0인데 1 → 부호 뒤집힘 Overflow
음수 + 음수 = 양수 MSB가 1+1인데 0 → 부호 뒤집힘 Overflow
양수 + 양수 = 양수 정상 OK
양수 + 음수 항상 범위 안 OK

뺄셈에서의 Overflow 🔥

  • 두 양수끼리, 두 음수끼리 뺄 때는 Overflow 불가능.
  • Overflow 발생 케이스 2가지:
    • 양수 − 음수 = 음수 → Overflow
    • 음수 − 양수 = 양수 → Overflow
  • 요약 공식: 부호가 다른 두 수의 뺄셈에서, 결과의 부호가 피감수(A)와 다르면 Overflow.

Overflow 기계적 검출법 🔥🔥🔥

MSB로 들어오는 Carry ≠ MSB에서 나가는 Carry → Overflow

  • 같으면 정상, 다르면 overflow.
  • HW에서 두 신호를 XOR만 하면 overflow 비트 바로 검출됨.

예제: 4비트 체계

  • 0111 (+7) + 0111 (+7):
    • 계산: 1110 (= −2 로 해석됨)
    • MSB 자리: 들어오는 carry=1, 나가는 carry=0 → 다름 → Overflow ✓
  • 1011 (−5) + 1100 (−4):
    • 계산: (1)0111 = +7 로 해석됨
    • MSB 자리: 들어오는 carry=0, 나가는 carry=1 → 다름 → Overflow ✓

Unsigned Integer vs Signed Integer 🔥

타입 Overflow 판정 기준
Unsigned MSB에서 Carry 발생 시 overflow
Signed MSB의 carry-in ≠ carry-out 시 overflow

3-2. MIPS와 Overflow 처리 🔥

핵심: MIPS는 NZCV 플래그 없음 ⚠️ 정정사항

  • 교수님이 ARM·x86과 헷갈려서 처음엔 NZCV가 있다고 설명했으나, MIPS는 NZCV 플래그 레지스터를 유지하지 않음.
  • 대신 연산 결과를 그때그때 즉시 확인하는 구조. (플래그 캐시 없음)
  • 참고: ARM/x86에는 있음.
    • N (Negative): 결과가 음수 → 1
    • Z (Zero): 결과가 0 → 1
    • C (Carry): MSB에서 carry 발생 → 1
    • V (oVerflow): overflow 발생 → 1

ARM / x86 분기 명령어 (참고용)

CPU Overflow 있을 때 분기 Overflow 없을 때 분기
ARM BVS (Branch if oVerflow Set) BVC (Branch if oVerflow Clear)
x86 JO (Jump if Overflow) JNO (Jump if No Overflow)

MIPS의 산술 명령어 종류 🔥

명령어 Overflow 처리
add, addi, sub Overflow 시 예외 발생
addu, addiu, subu Overflow 무시
  • 왜 unsigned 버전은 overflow를 무시하나? 🔥
    • 덧셈이 항상 산술 목적은 아님. 주소 계산, 배열 인덱스, 스택 포인터 이동, 비트 패턴 등 overflow가 발생해도 상관없는 연산이 많음.
    • → 속도와 편의성을 위해 addu 계열은 예외 발생 안 함.

3-3. 예외(Exception) vs 인터럽트(Interrupt) 🔥🔥

둘 다 "CPU의 정상 흐름을 가로채는" 이벤트지만 발생 원인이 다름.

구분 예외 (Exception) 인터럽트 (Interrupt)
발생 원인 현재 실행 중인 명령어 내부 문제 외부 장치로부터의 신호
예시 Overflow, Divide by 0, Invalid opcode 키보드 입력, 네트워크 패킷 도착, 타이머
동기성 동기적 (명령어와 직접 연관) 비동기적

예외 처리 흐름 🔥

  1. 현재 실행 중 명령어의 PC 값을 EPC (Exception PC) 레지스터에 저장 ← 복귀용
  2. 미리 정의된 Exception Handler 주소로 점프 (PC ← Handler 주소)
  3. 예외 처리 루틴 수행
  4. EPC 값을 PC에 복원 → 원래 흐름으로 복귀 (eret 명령어)

Interrupt 처리 핵심 포인트

  • CPU는 Fetch → Decode → Execute를 반복하는 "멍청한 놈".
  • Execute 후, 다음 Fetch 전에 인터럽트 핀(pin)을 확인.
  • 인터럽트 발생 시 PC를 인터럽트 핸들러 주소로 바꿔치기.
  • 우선순위(priority) & 중첩(nested): 처리 중 더 높은 우선순위의 인터럽트가 오면 중첩 처리 가능.
  • 인터럽트는 무시 불가 (네트워크 패킷을 그냥 버릴 수는 없으니까).

3-4. 코프로세서 0 (CP0) 🔥

MIPS CPU에 보조적으로 붙어 있는 시스템 제어 전용 프로세서.

역할

  1. 예외 / 인터럽트 처리 지원 (예외 원인, 핸들러 분기 등)
  2. CPU 상태 제어 (인터럽트 Enable/Disable, 모드 전환)
  3. TLB 관리 (가상 메모리 변환 버퍼)
  4. 시스템 레지스터 제공 — Status, Cause, EPC 등

CP0 주요 레지스터

번호 이름 용도
12 Status CPU 상태 / 인터럽트 Enable 플래그 등
13 Cause 예외 발생 원인
14 EPC 예외 발생 시점의 PC (복귀 주소)

레퍼런스 카드 기준 비트 필드까지 같이 보기

레지스터 중요 비트 의미
Status 15:8 Interrupt Mask - 어떤 인터럽트를 허용할지 선택
Status 4, 1, 0 UM (User Mode), EL/EXL (예외 레벨), IE (Interrupt Enable)
Cause 31 BD - Branch Delay Slot에서 예외가 났는지 표시
Cause 15:8 Pending Interrupt - 현재 걸려 있는 인터럽트
Cause 6:2 Exception Code - 예외 종류 번호

즉 시험에서 "예외가 왜 났는지"는 Cause, "복귀할 주소가 어디인지"는 EPC, "인터럽트를 받을 수 있는 상태인지"는 Status로 본다고 정리하면 됩니다.

CP0 전용 명령어 🔥

명령어 의미
mfc0 $rt, $rd Move From Coprocessor 0. CP0의 $rd → 일반 레지스터 $rt
mtc0 $rt, $rd Move To Coprocessor 0. 일반 레지스터 $rt → CP0의 $rd
eret Exception RETurn. EPC → PC (복귀)

예)

mfc0  $k0, $14        # EPC 값을 $k0로 가져오기
mtc0  $t0, $12        # $t0 값을 Status 레지스터에 쓰기
eret                  # 예외 처리 종료, 원래 위치로 복귀

TLB (Translation Lookaside Buffer) 간단 정리 🔥

가상 메모리 주소 → 물리 메모리 주소 변환 캐시.

  • 가상 메모리 문제: 프로그램 100GB인데 RAM은 32GB. 전부 올릴 수 없음 → 가상 메모리 사용.
  • 주소 변환에 페이지 매핑 테이블 필요 → DRAM에 있음 → 매번 읽으면 메모리 접근 2회 (테이블 읽기 + 실제 데이터 읽기) 필요 → 너무 느림.
  • TLB: CPU 내부에 위치한 최근 변환 정보 캐시. 지역성 덕에 잘 동작.
  • Context Switch 시: 이전 프로세스의 TLB는 무효 → Flush(초기화) 필요.

레퍼런스 카드에 나온 대표 Exception Code

번호 이름 의미
0 Int 하드웨어 인터럽트
4 AdEL 주소 오류 (load 또는 instruction fetch)
5 AdES 주소 오류 (store)
8 Sys syscall 예외
9 Bp breakpoint
10 RI 예약/미정의 명령어
11 CpU 지원되지 않는 coprocessor 사용
12 Ov 산술 overflow
13 Tr trap
15 FPE Floating Point Exception

전부 다 달달 외우라는 뜻은 아니고, 주소 오류 / syscall / reserved instruction / overflow 정도는 바로 떠올라야 문제 풀이가 편합니다.

3-5. 곱셈 — 기본 원리 🔥

용어

  • Multiplicand (피승수): 곱해지는 수 (예: 8 × 9 에서 8)
  • Multiplier (승수): 곱하는 수 (예: 8 × 9 에서 9)
  • Product (곱, 결과)

2진수 곱셈 규칙

승수의 현재 자리가 1이면 피승수를 그 자리만큼 Shift해서 더함. 0이면 아무것도 안 함.

  • n비트 × n비트는 최대 2n비트 결과를 가짐.
  • 32비트 × 32비트 = 64비트 필요.

Array Multiplier (병렬 곱셈기)

  • AND 게이트로 각 비트별 부분곱(Partial Product) 동시 생성.
  • Full Adder로 각 열의 합 + Carry 처리.
  • 장점: HW로 구현 용이, 병렬 처리 가능.
  • 단점: 32비트 기준 AND 게이트 32×32=1024개, Adder도 산더미 → HW 복잡도 高.

3-6. 곱셈 HW 3세대 진화 🔥🔥🔥 시험 필출

공통 동작 규칙

  • 승수의 LSB가 1 → Product에 Multiplicand 더함
  • 승수의 LSB가 0 → 아무것도 안 함
  • 32회 반복 (32비트 기준)

Version 1 — 원시 버전

┌───────────────────────────────┐
│ Multiplicand (64-bit, shift L)│
└───────┬───────────────────────┘
        ↓
      ┌───┐
      │ALU│ 64-bit
      └─┬─┘
        ↓
┌───────────────────────────────┐      ┌──────────────────┐
│      Product (64-bit)         │ ←──  │ Multiplier       │
└───────────────────────────────┘      │ (32-bit, shift R)│
                                        └──────────────────┘
  • Multiplicand: 64비트 레지스터 + 왼쪽 Shift (자리수 이동용).
  • Multiplier: 32비트 레지스터 + 오른쪽 Shift (LSB 확인용).
  • ALU: 64비트.
  • 한 단계 = 3 서브 스텝 (ALU + Multiplicand Shift + Multiplier Shift).

Version 2 — ALU 폭 축소 🔥

┌─────────────────┐
│ Multiplicand    │
│    (32-bit,     │
│     고정)       │
└────┬────────────┘
     ↓
   ┌───┐
   │ALU│ 32-bit
   └─┬─┘
     ↓ (상위 절반에만 더함)
┌───────────────────────────────┐      ┌──────────────────┐
│   Product (64-bit, shift R)   │      │ Multiplier       │
└───────────────────────────────┘      │ (32-bit, shift R)│
                                        └──────────────────┘
  • Multiplicand 고정 (Shift 안 함).
  • 대신 Product를 오른쪽으로 Shift. 자리수 맞추기는 Product 쪽에서.
  • ALU도 32비트로 축소 → HW 간소화.

Version 3 — 최종판 🔥🔥

┌─────────────────┐
│ Multiplicand    │
│   (32-bit)      │
└────┬────────────┘
     ↓
   ┌───┐
   │ALU│ 32-bit
   └─┬─┘
     ↓ (상위 절반에만 더함)
┌───────────────────────────────┐
│   Product (64-bit, shift R)   │
│   [상위: 결과 누적]           │
│   [하위: Multiplier 저장]     │ ← 별도 Multiplier 레지스터 불필요!
└───────────────────────────────┘
  • 핵심 통찰: Product는 초기에 하위 절반이 0으로 비어있음 → 거기에 Multiplier를 넣자.
  • Product를 오른쪽으로 Shift하면 하위 비트가 차례로 LSB로 이동 → Multiplier의 각 비트를 순차 확인 가능.
  • 별도 Multiplier 레지스터 제거 → HW 추가 축소.
  • 한 단계 = 2 서브 스텝 (ALU + Product Shift R).

3세대 비교표

버전 Multiplicand Product Multiplier ALU 비고
V1 64-bit, ShiftL 64-bit 32-bit, ShiftR 64-bit 가장 단순, HW 과다
V2 32-bit, 고정 64-bit, ShiftR 32-bit, ShiftR 32-bit ALU 축소
V3 32-bit, 고정 64-bit, ShiftR (Multiplier 포함) 없음 32-bit 최종형

시험 시나리오: 블록 다이어그램 주고 "중간 단계 값 채워라", "어느 버전인지 식별해라" 나올 수 있음.

3-7. 부호 있는 곱셈 (Signed Multiplication) 🔥

처리 절차

  1. 입력값의 부호 확인 (MSB 확인).
  2. 음수면 2의 보수 해제하여 절댓값(양수)로 변환.
  3. 양수끼리 곱셈 수행 (앞의 Unsigned 알고리즘).
  4. 최종 부호 결정: 피승수 부호 ⊕ 승수 부호 (XOR)
    • (양, 양) → 양
    • (양, 음) or (음, 양) → 음
    • (음, 음) → 양
  5. 결과가 음수여야 하면 2의 보수 적용.

3-8. 더 빠른 곱셈기 (Pipelined Multiplier) 🔥

  • 부분곱 생성 자체는 병렬화 가능 (AND 게이트로 한 방에).
  • Adder 계층(Reduction Tree)을 파이프라인 스테이지로 분할.
  • N개 스테이지 → 매 클럭마다 새로운 곱셈 시작 가능 → 처리량(throughput) 극대화.

"곱셈 연산은 여러 개의 곱셈을 병렬로 수행하는 것도 가능"

3-9. MIPS 곱셈 명령어 🔥🔥

Hi / Lo 레지스터

  • 64비트 결과를 Hi(상위 32비트) + Lo(하위 32비트) 에 저장.
  • 일반 레지스터(R0~R31)와 별개.
  • 직접 접근 불가 → mfhi, mflo로 가져와야 함.

명령어

명령어 의미
mult $s2, $s3 Hi:Lo ← $s2 × $s3 (signed)
multu $s2, $s3 Unsigned 곱셈
mfhi $t0 $t0 ← Hi
mflo $t0 $t0 ← Lo

의사 명령어 mul

mul  $t0, $s2, $s3
# == mult $s2, $s3 + mflo $t0
  • 하위 32비트만 가져옴. 결과가 32비트 초과하면 상위 비트 손실 → 주의!

3-10. 나눗셈 — 기본 원리 🔥

용어

  • Dividend (피제수): 나누어지는 수
  • Divisor (제수): 나누는 수
  • Quotient (몫)
  • Remainder (나머지)

공식

Dividend = Divisor × Quotient + Remainder

복원 나눗셈 (Restoring Division) — 기본형

  1. Remainder 레지스터에 Dividend 초기 세팅.
  2. Remainder를 왼쪽으로 1비트 Shift.
  3. Remainder에서 Divisor를 뺀다.
  4. 결과가 음수면: Divisor를 다시 더해 복원, 현재 몫 비트를 0.
  5. 결과가 양수(또는 0)면: 그대로 두고, 현재 몫 비트를 1.
  6. n번 반복 (n비트 기준).

핵심 문제: 음수 나올 때마다 복원 연산(더하기) 이 추가됨 → 비효율.

비복원 나눗셈 (Non-Restoring Division) 🔥🔥

논리적 치환: 복원하고 다음에 빼느니, 복원 안 하고 다음에 더하자.

증명 (식 전개)

  • 복원 + Shift + 뺄셈: 2 × (R + D) − D = 2R + 2D − D = 2R + D
  • 비복원 + Shift + 덧셈: 2R + D
  • → 완전히 동일.

알고리즘

  1. Remainder Shift Left.
  2. 이전 단계 Remainder가 음수였으면 → Divisor 더함. 이전 단계 Remainder가 양수였으면 → Divisor 뺌.
  3. 결과가 0 이상 → 몫 비트 1. 결과가 음수 → 몫 비트 0. (복원 X)
  4. n번 반복.
  • 장점: 복원 단계 제거 → 속도 향상.

나눗셈 블록 다이어그램 개념

┌──────────────────────┐
│ Remainder (2n-bit)   │ ← shift L
│ [상위: 실제 나머지]  │
│ [하위: 몫 저장]      │
└──────┬───────────────┘
       ↓ (상위 절반)
     ┌───┐
     │ALU│ (뺄셈 또는 덧셈)
     └─┬─┘
       ↑
┌──────────────┐
│ Divisor      │
└──────────────┘

곱셈 V3와 비슷: Remainder 하위 절반에 몫(Quotient)이 순차 채워짐.

3-11. MIPS 나눗셈 명령어 🔥

명령어 의미
div $s2, $s3 Hi ← Remainder, Lo ← Quotient (signed)
divu $s2, $s3 Unsigned 나눗셈
mfhi $t0 나머지 꺼내기
mflo $t0 몫 꺼내기
  • Divide by 0 체크 안 함 → 프로그래머가 직접 확인해야 함.

3-12. 곱셈 vs 나눗셈 속도 🔥

연산 사이클 수 비고
Add / Shift 1 사이클 가장 빠름
곱셈 수 사이클 파이프라인으로 throughput ↑
나눗셈 수십 사이클 순차적, 병렬화 어려움
  • 특정 아키텍처의 문제가 아니라 연산 구조 자체의 차이.
  • 암달의 법칙: 나눗셈이 느려도 빈도가 적으면 전체 성능에 큰 영향 없음.

3-13. 부동소수점 (Floating Point) — 개념 🔥🔥

왜 필요한가?

  • 정수만으론 부족. float, double 타입이 이것.
  • 한계: 모든 실수를 정확히 저장 불가. 예) 0.1조차 2진수로는 무한소수.

과학적 표기법 (Scientific Notation) vs 정규화된 표현 (Normalized)

표기법 조건 예시
과학적 표기법 a × Bᵉ 꼴 31.5576 × 10⁸, 0.315576 × 10¹⁰
정규화된 표현 위 + 맨 앞자리 ≠ 0. 10진수는 1 ≤ |a| < 10 3.15576 × 10⁹

왜 정규화? → 표기 유일성 보장 + 계산이 쉬워짐 (곱셈 시 a₁×a₂, e₁+e₂로 분리 가능).

2진수 정규화 특수성 🔥

  • 2진수 정규화 → 맨 앞은 항상 1. (0이 아니어야 하므로)
  • → 맨 앞의 1은 저장 안 함 (Hidden Bit, 암묵적 1).
  • 실제 저장은 소수점 이하만.

MIPS 부동소수점 명령어 패밀리 (레퍼런스 카드 보강)

분류 대표 명령어 설명
산술 add.s, sub.s, mul.s, div.s
add.d, sub.d, mul.d, div.d		 .s=single precision, .d=double precision
비교 c.eq.s, c.lt.s, c.le.s
c.eq.d, c.lt.d, c.le.d		 비교 결과는 일반 레지스터가 아니라 FP condition에 반영
분기 bc1t, bc1f FP condition이 참/거짓일 때 분기
메모리 lwc1, swc1, ldc1, sdc1 부동소수점 레지스터와 메모리 사이 이동

레퍼런스 카드에는 부동소수점 전용 형식도 따로 실려 있습니다. 단정도/배정도 연산은 일반 정수 R-type과 비슷해 보여도 필드 이름이 다르다는 점이 중요합니다.

FR format : opcode | fmt | ft | fs | fd | funct
FI format : opcode | fmt | ft | immediate

3-14. IEEE 754 포맷 🔥🔥🔥 시험 필출

정규화된 IEEE 754 값 해석 공식

(-1)^S × (1 + Fraction) × 2^(Exponent - Bias)
  • Single Precision의 Bias는 127, Double Precision의 Bias는 1023.
  • 여기서 1 + Fraction의 맨 앞 1은 hidden bit. 정규화된 수에서는 저장하지 않지만 존재한다고 가정한다.
  • 단, 비정규화 수(denormal)는 이 공식을 그대로 쓰지 않고 숨은 1이 0.xxxxx 형태로 바뀐다.

단일 정밀도 (Single Precision, 32bit) — 반드시 암기

┌────┬────────────────────┬────────────────────────────────────────────┐
│ S  │    Exponent        │              Fraction                      │
│1bit│      8 bit         │              23 bit                        │
└────┴────────────────────┴────────────────────────────────────────────┘
 31    30              23 22                                         0
  • S (Sign): 1bit. 0=양수, 1=음수.
  • Exponent: 8bit. Bias = 127 적용.
  • Fraction: 23bit. 소수점 이하 저장 (Hidden Bit 제외).

정밀도별 정리

정밀도 총 비트 지수 소수 Bias
Half 16 5 10 15
Single 32 8 23 127
Double 64 11 52 1023
Quadruple 128 15 112 16383

Bias 표현 왜 쓰나? 🔥

  • Exponent 자리에 2의 보수를 쓰면 대소 비교가 꼬임 (음수가 더 크게 보임).
  • → 단순 비교기(comparator)로 실수 대소 비교하기 위해 Bias 도입.
  • Single Precision Bias = 127 → 실제 지수 e 저장값 = e + 127.
    • 실제 지수 −126 → 저장값 1
    • 실제 지수 +127 → 저장값 254

대소 비교 최적화 설계 🔥

  • 포맷 순서 S | Exp | Frac 는 우연이 아님.
  • 비교 순서: ① 부호 → ② 지수 → ③ 소수.
  • 지수를 소수 앞에 둔 이유: 지수만 크면 더 큰 값이므로.

Exponent 값의 특수 의미 🔥🔥

Exponent Fraction 의미
0 0 0.0 (양수/음수 부호 따로)
0 ≠ 0 비정규화 수 (denormalized)
1 ~ 254 임의 정규화된 실수
255 0 ±∞ (Infinity)
255 ≠ 0 NaN (Not a Number)

레퍼런스 카드 표로 다시 요약

  • Exponent = 0, Fraction = 0 → ±0
  • Exponent = 0, Fraction ≠ 0 → ±Denorm
  • Exponent = 1 ~ MAX-1 → 정규화 수
  • Exponent = MAX, Fraction = 0 → ±∞
  • Exponent = MAX, Fraction ≠ 0 → NaN

여기서 MAX는 single에서 255, double에서 2047입니다.

NaN (Not a Number) 🔥

  • 수학적으로 정의되지 않는 연산의 결과.
  • 예) 0/0, ∞−∞, log(−1), NaN + 2, NaN × 2 — 모두 NaN.
  • C언어: float x = NAN;, printf("%f", x) → nan 출력.
  • Python: float('nan').

Overflow / Underflow in Floating Point 🔥

  • Overflow: 표현 가능한 최댓값 초과. 양/음 방향 모두 가능.
  • Underflow: 값이 0에 너무 가까워 표현 불가.

MIPS Single Precision 표현 범위

음수 Overflow   표현가능 음수   Underflow   표현가능 양수   양수 Overflow
    │                │          │   0   │         │                │
─────┤────────────────┤──────────┤───────┤─────────┤────────────────├─────
  −∞ │  −2×10³⁸       │ −2×10⁻³⁸ │       │ 2×10⁻³⁸ │     2×10³⁸     │ +∞
  • 대략 ±2 × 10³⁸ 범위. 숫자 2, 38 기억.

3-15. IEEE 754 변환 실전 🔥🔥🔥 시험 필출

10진 소수 → 2진 소수 변환

소수 파트에 2를 계속 곱하면서 정수부를 기록.

예) 0.625 → ?

단계 계산 정수부
1 0.625 × 2 = 1.25 1
2 0.25 × 2 = 0.5 0
3 0.5 × 2 = 1.0 1

→ 0.625₁₀ = 0.101₂ (= ½ + ⅛)

0.1은 2진수로 무한소수 🔥

  • 0.1 → 0.2 → 0.4 → 0.8 → 1.6 → 1.2 → 0.4 → ... 패턴 반복.
  • 10진 0.1 = 2진 0.0001100110011... (무한)
  • → Floating Point로 0.1은 정확 저장 불가. 항상 근사값.

10진 실수 → IEEE 754 Single Precision 변환 절차 🔥

예제: −0.75 → IEEE 754 32bit

Step 1. 2진수로 변환

  • 0.75₁₀ = 0.11₂ (½ + ¼)

Step 2. 정규화

  • 0.11₂ = 1.1 × 2⁻¹

Step 3. 각 필드 결정

필드 값 비트
S (부호) 음수 1
Exponent −1 + 127 = 126 01111110
Fraction 소수점 이하만 (Hidden Bit 제외) → 1 + 22개의 0 10000000000000000000000

Step 4. 완성된 32bit

1 | 01111110 | 10000000000000000000000

Step 5. 16진수 변환

비트를 4개씩 묶어서:

1011 1111 0100 0000 0000 0000 0000 0000
  B    F    4    0    0    0    0    0

→ 0xBF400000

역변환: 16진수 → 10진 실수도 가능해야 함 🔥

  • 주어진 32비트를 S / Exp / Frac로 쪼갬.
  • (−1)^S × 1.Fraction × 2^(Exp − 127) 계산.

실수자 포인트 🔥⚠️

  • Hidden Bit를 Fraction에 또 쓰는 실수 금지. 소수점 왼쪽의 1은 저장 안 함.
  • Bias 잊지 말기. −126을 저장하려면 Exponent 필드에 1 이 들어감 (−126 + 127 = 1).
  • 2의 보수 형태가 아님 — Exponent는 Biased.
